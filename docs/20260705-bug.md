## Bug 1：成就徽章逻辑 —— 新用户会瞬间解锁全部徽章

**严重级别**：高（影响真实使用）

**位置**：`SmokeFree/SmokeFree/SmokeFree/Services/AchievementService.swift`
- `consecutiveDaysBelow(baseline:logs:)` （约 61 行）
- `sevenDayAverage(logs:)` （约 82 行）

**现象**：
`consecutiveDaysBelow` 固定回溯 365 天，且把"某天没有记录"一律视为"0 支 = 低于基准"继续计数，回溯**没有以 `quitDate` 为下界**。

**后果（具体例子）**：
一个今天刚开始戒烟的新用户，记录 1 条低于基准的日志后，`consecutiveDaysBelow` 返回 **365** → 一次性解锁全部连续天数徽章（1 天…365 天）；同时 `sevenDayAverage` 对空缺天计 0，也会同时满足减半 / 接近清零两个减量徽章。`LoggingView` 在每次记录后都会调用 `evaluateAndAward`，因此真实使用中会触发。

**已有测试可复现（当前为失败/已 disable，指向本 Bug）**：
- `AchievementServiceTests.noAwards_whenStreakIsZero`
- `AchievementServiceTests.noStreak3Days_whenOnlyOneDayBelow`
- `AchievementServiceTests.awards_multipleStreakBadges_when7DayStreak`（断言不应包含 30 天徽章）

这些测试此前从未运行（工程无测试 target），因此 Bug 长期未被发现。

**候选修复方案（需先定语义）**：
- A1：回溯以 `quitDate` 为下界（对齐 `UserProfile.actualStreakDays`）。副作用：新用户第 1 天仍会拿到"1 天"徽章。
- A2：无记录的那天直接中断连续计数（最严格，最贴合上述 3 个测试意图）。
- A3：仅修 `consecutiveDaysBelow` 固定 365 回溯，改为止于最早日志 / quitDate。

**修复完成后**：重新启用上述 3 个 `@Test(.disabled(...))`，并确认全绿。

## Bug 2：金额使用 Double，违反"金额一律用 int（分）"规则

**严重级别**：中（架构 / 精度隐患，非当前崩溃）

**范围**：全 App 级重构（100+ 处），涉及：
- Model：`UserProfile.pricePerPack` / `goalAmount`、`SmokingLog.pricePerPackAtTime`、`PurchaseRecord.pricePerPack` / `totalCost`、`Goal.targetMoneySaved`
- Core Data schema：`SmokeFreeApp.swift` 中 `makeModel()` 对应 6 个 `.doubleAttributeType`
- 所有相关 ViewModel（Dashboard/Purchase/Goals/Charts 等）金额计算
- 导入导出：`DataExportService` / `DataImportService`
- Widget：`WidgetProvider` / `SmokeFreeWidget`
- 成本对比预设：`AppConfig.costComparisonPresets`

**关键风险 —— Core Data 迁移必须保数据**：
把 `Double` 直接改为 `Int64`，Core Data 轻量迁移会把 `25.0` 变成 `25` 分（= ¥0.25），**数据损坏**。开工前必须先确认：

- 若无真实用户数据（未上架 / 可清库）→ 直接改 schema 类型 + 清一次本地库，简单快。
- 若有真实用户数据 → 新增"分"字段并列，首次启动做一次性迁移（读旧 Double ×100 写新 Int），保留旧字段，复杂但不丢数据。

**建议**：单独 session 用 /design 流程处理，先定迁移策略再动手。
