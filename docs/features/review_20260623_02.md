# Review 02: 全量功能实现逻辑审查

Date: 20260623
Focus: 业务逻辑正确性 — 每个功能"应该做什么" vs "实际做了什么"

---

## 总览

审查发现 **5 类系统性设计缺陷**，根源相同：多个功能只依赖"日历时间流逝"，不检查用户是否真的在控烟。

```
时间驱动（当前实现）         行为驱动（应该实现）
─────────────              ─────────────
quitDate → 日历天数 → 里程碑解锁   quitDate + 每日记录 → 实际控烟天数 → 里程碑解锁
quitDate → 日历天数 → 通知推送    quitDate + 每日记录 → 达标才推送
quitDate → 日历天数 → Widget     quitDate + 每日记录 → 实际天数和金额
```

## 功能 1: 控烟里程碑时间线

### 1.1 解锁条件只看日历 [Critical]

**文件:** `HealthTimelineViewModel.swift:17-18`, `AppConfig.swift:78-121`

```swift
// HealthTimelineViewModel.load():
let elapsed = now.timeIntervalSince(quitDate)
let isUnlocked = elapsed >= milestone.offsetSeconds  // ← 只看时间
```

**反例:** 用户 100 天前设定 quitDate，之后每天抽 20 支（等于基准）。100 天后：
- 首页连续控烟天数显示 **0 天**（正确）
- 但时间线页解锁了"三个月荣耀""半年里程碑"等全部里程碑（**错误**）
- 里程碑文案说"肺功能显著改善，精力更充沛"——用户根本没有控烟

**根因:** `isUnlocked` 的逻辑假设了"设定 quitDate = 控烟开始"，但实际需要验证用户在此期间是否真的低于基准。

### 1.2 己解锁里程碑永久保持 [Major]

即使后续用户回到基准甚至超标，已解锁的里程碑永远显示为"已完成"。没有"复发重置"概念。控烟应用不应该把"三个月荣耀"永久授予一个已经放弃控烟的人。

### 1.3 Dashboard 首页里程碑进度同样只看时间

**文件:** `DashboardViewModel.swift:22-37`

```swift
let elapsed = profile.smokeFreeSeconds  // 纯日历时间
let next = milestones.first { $0.offsetSeconds > elapsed }
```

首页显示的"下个控烟里程碑"和进度环与用户实际行为完全脱节。

## 功能 3: 连续控烟天数

### 3.1 首页 streak（actualStreakDays）vs 成就 streak（consecutiveDaysBelow）逻辑不一致 [Major]

| 场景 | actualStreakDays（首页） | consecutiveDaysBelow（成就） |
|------|--------------------------|------------------------------|
| 无记录 | 计入（宽松） | 中断（严格） |
| 今天无记录 | 从今天开始往前数，今天无记录算 1 天 | 检查今天无记录 → 立即中断 → 返回 0 |

**影响:**
- 首页显示"连续控烟 N 天"，但成就页一个 streak 徽章都不解锁 → 用户困惑
- 如果今天还没记录，`consecutiveDaysBelow` 一定返回 0 → 所有 streak 成就今天都不可能解锁

**文件:** `UserProfile.swift:48-65`, `AchievementService.swift:59-78`

### 3.2 actualStreakDays 对"无记录"的处理可被利用

用户只需停止记录（不保存任何 log），就能维持 streak 增长。实际可能每天抽 25 支但从不记录。

### 3.3 两个 streak 计算中 `first(where:)` 的 O(n²) 问题只是性能问题，真正的问题是"无记录"的语义分歧

## 功能 4: 节省金额计算

### 4.1 moneySaved 无除零保护 vs totalSpentOnSmokes 有保护 [Major]

**文件:** `UserProfile.swift:74` vs `DashboardViewModel.swift:88`

```swift
// moneySaved — 无保护
return sum + Double(reduced) * (price / Double(perPack))  // perPack 可为 0

// totalSpentOnSmokes — 有保护
let perPack = max(1, log.cigarettesPerPackAtTime ?? profile.cigarettesPerPack)  // min 1
```

如果用户修改 `cigarettesPerPack` 为 0，`moneySaved` 路径直接 crash；`totalSpentOnSmokes` 不会。

### 4.2 moneySaved 对所有日志全局求和，无时间窗口概念

如果用户 3 年前控烟 7 天省了 ¥100，后来又恢复了，那些旧日志节省的钱永久算在"已节省"里。

### 4.3 每日记录数量巨大时，函数从零求和效率低

每次调用遍历全部 logs 从头累加，无缓存。但语义正确（实际少抽支数 × 单支价格）。

## 功能 5: 目标系统

### 5.1 目标完成使用 actualStreakDays（宽松），与目标名称"连续控烟 N 天"语义冲突

**文件:** `GoalsViewModel.swift:31-44`, `GoalsView.swift:73-81`

目标标题是"连续控烟 7 天"，但 `checkCompletion` 使用 `actualStreakDays`：
- 如果用户 7 天前设定了目标，期间从不记录 → 7 天后自动完成 ✓/✗？
- 如果用户每天都记录但有一天记录 == 基准 → streak 中断，目标不完成 ✓
- 如果用户 7 天都记录且 < 基准 → streak = 7，目标完成 ✓

**边界问题:** 未来某天的记录不会影响今天的 streak 判断。但用户可能昨天记录超标，streak 昨天已中断。今天目标检查时 streak 可能是 0。这不是 bug。

### 5.2 金额目标完成：checkCompletion 被触发但 moneySaved 此时可能因无记录而不更新

`GoalsView.checkCompletion()` 被 `onChange(of: logs)` 触发，但 moneySaved 只在有新日志时才变化。如果用户记录了一条新日志，moneySaved 可能增加足够完成目标。逻辑正确。

### 5.3 checkCompletion 被 onChange(of: goals) 自身触发可能递归

`checkCompletion()` 修改 `goal.isCompleted = true` → SwiftData 触发 `onChange(of: goals)` → `checkCompletion()` 再次调用 → 但 `where !goal.isCompleted` 守卫阻止了死循环。安全。

## 功能 6: 成就系统

### 6.1 成就评估在 DashboardView 每次更新时触发，但使用 `logs`（`@Query` 的参考）

**文件:** `DashboardView.swift:93`

```swift
AchievementService.evaluateAndAward(profile: profile, logs: logs, context: context)
```

`logs` 是 `SmokingLog` 数组（不是 `Array(logs)` 拷贝），直接传入。`evaluateAndAward` 内部执行 `context.fetch()` 查询已解锁成就，然后 `context.insert()` 新成就。在 `@Query` 结果上做 mutation 可能有问题（insert 不会修改传入的 logs 数组，但 context 变化会触发 @Query 重新 fetch，导致 onChange 触发，形成二次评估）。有去重逻辑（`unlockedIDs` 检查），所以安全但可能浪费。

### 6.2 LoggingView 保存后评估使用旧数据

**文件:** `LoggingView.swift:26-28`

```swift
vm.save(context: context, profile: profiles.first)
AchievementService.evaluateAndAward(profile: profile, logs: Array(logs), context: context)
```

`save()` 内部 `context.insert(log)` 后，`logs`（`@Query` 结果）在 save 返回时可能尚未更新。导致今天保存的记录不被计入本次成就评估。不过 `DashboardView.onChange(of: logs)` 之后会重新评估，所以成就最终会被解锁。

### 6.3 减量幅度徽章使用"近 7 天日均"，分母固定为 7

**文件:** `AchievementService.swift:82-91`

```swift
let total = (0..<7).reduce(0) { sum, offset in
    // ...
    let count = logs.first(where: { $0.date == date })?.count ?? 0  // 无记录 = 0 支
    return sum + count
}
return Double(total) / 7.0  // 固定除 7
```

如果用户只有 3 天有记录，那 4 天空白计为 0。这意味着 `sevenDayAverage` 倾向于偏低，减量成就更容易解锁。对于"减半达人"（需 ≤ 50% 基准），如果只记录了 2 天且都是 0 支，就能解锁。这是否合理取决于产品意图。

### 6.4 两个"超过基准"的情况被视为"今天多了一点"但不影响已解锁成就

已解锁的 `streak_1_day` 等成就不会被撤销。一旦拿到就永久保留。这与 streak 概念矛盾（streak 应是持续状态）。

## 功能 7: 首页 Dashboard

### 7.1 Dashboard 显示 5 张卡片但缺少"下个里程碑"卡片

`HealthStatusCardView` 在 `DashboardView.swift:359` 定义了但从未被使用。首页只显示了 StreakCard、ReductionCard、MoneySavedCard、SmokingCostCard。缺少里程碑进度卡片。

### 7.2 Timer 每分钟触发全量更新

**文件:** `DashboardView.swift:82-84`

```swift
.onReceive(Timer.publish(every: 60, on: .main, in: .common).autoconnect()) { _ in
    updateVM()  // 调用 actualStreakDays, moneySaved, evaluateAndAward, HealthKit
}
```

每分钟：遍历全部日志计算 streak + 遍历全部日志计算 moneySaved + 查询全部已解锁成就 + 检查全部成就定义 + 可能写 HealthKit。电池和性能不友好。且切换到其他 Tab 时 Timer 仍在运行。

### 7.3 updateVM 每次都尝试写 HealthKit

**文件:** `DashboardView.swift:94-96`

```swift
if vm.todayCount >= 0 && vm.todayCount < vm.baselineCount {
    Task { await HealthKitService.shared.recordSmokeFreeToday() }
}
```

HealthKitService 有去重（检查今天是否已写入），但每分钟触发一次这个检查也是浪费。

### 7.4 "尚未记录"时 todayCount = -1 的语义

`DashboardViewModel.todayCount = -1` 表示"今天尚未记录"，这是合理的 sentinel 值。但 HealthKit 判断 `todayCount >= 0 && todayCount < vm.baselineCount` 使用了这个 sentinel 作为守卫。正确。

## 功能 8: HealthKit 集成

### 8.1 "控烟"写入"正念记录"的语义不匹配 [Minor]

**文件:** `HealthKitService.swift:56-62`

App 在用户"低于基准"时写入一个 Mindful Session 到 Health App。但 mindful session 在 Apple Health 中的语义是"正念/冥想"，不是"控烟/减烟"。用户在 Health App 中看到的是"正念分钟数"而不是控烟数据。这是 Apple HealthKit 类型选择的限制，没有更好的类型可用，但至少应写入注释说明这个 trade-off。

### 8.2 写入条件为 `todayCount < baselineCount` 而非 `todayCount == 0`

每天抽 19 支（基准 20 支）也写入"无烟正念记录"。这个条件与 `recordSmokeFreeToday` 方法名矛盾。

## 功能 9: Widget 数据桥接

### 9.1 Widget 显示的 streakDays 是 actualStreakDays（与首页一致）

**文件:** `DashboardViewModel.swift:107-115`

`writeWidgetData` 写入 `vm.streakDays`（来自 `actualStreakDays`，宽松）。Widget 和首页显示的 streak 一致。正确。

### 9.3 Widget 刷新策略为每小时

**文件:** `WidgetProvider.swift:47`

但 `WidgetCenter.shared.reloadAllTimelines()` 会强制刷新。所以实际行为是：每次切到首页 → 立即刷新 Widget + 每小时自动刷新。

## 功能 10: 图表系统

### 10.1 日均计算只统计"有记录的天"

**文件:** `ChartsViewModel.swift:58-60`

```swift
let recordedDays = cigaretteData.filter { logByDate[$0.date] != nil }
avgCigarettes = recordedDays.isEmpty ? 0 :
    Double(recordedDays.map(\.count).reduce(0, +)) / Double(recordedDays.count)
```

如果 7 天中只有 2 天有记录（各 10 支），日均显示为 10 支。但更准确的做法应是 `总烟量 / 7`，否则短线数据可能误导用户以为日均很低。不过 DESIGN.md 说"无记录天计为 0，分母固定为 7"用于成就评估（`sevenDayAverage`），图表采用了不同策略。

### 10.2 图表基准减少百分比计算

**文件:** `TrendsView.swift:40-48`

```swift
let pct = vm.avgCigarettes < Double(vm.baselineDailyCount)
    ? Int((1 - vm.avgCigarettes / Double(vm.baselineDailyCount)) * 100)
    : 0
```

当 `avgCigarettes >= baselineDailyCount` 时显示 0% 而非负数。合理。

### 10.3 TrendsView 无数据空态

**文件:** `TrendsView.swift`

当日志全部为空时，图表全零柱子，没有任何引导文字。应有空态处理。

## 总结：核心架构问题

### 日历时间与行为数据的脱节（影响功能 1, 2, 9）

| 模块 | 数据源 | 是否正确反映控烟行为 |
|------|--------|---------------------|
| 首页 StreakCard | actualStreakDays(logs:) | ✅ 基于实际记录 |
| 首页 里程碑进度 | smokeFreeSeconds（纯时间） | ❌ 与记录无关 |
| 时间线页 里程碑 | smokeFreeSeconds（纯时间） | ❌ 与记录无关 |
| 通知 里程碑 | quitDate + 时间偏移 | ❌ 与记录无关 |
| Widget streakDays | actualStreakDays（via writeWidgetData） | ✅ 与首页一致 |
| 目标进度 | actualStreakDays(logs:) | ⚠️ 宽松，无记录算达标 |
| 成就 streak | consecutiveDaysBelow | ✅ 严格，但今天无记录必为 0 |

### 建议修复优先级

1. **里程碑系统改为行为驱动** — 解锁条件从纯时间改为 `actualStreakDays >= milestone对应天数`
2. **通知系统同步修改** — 里程碑通知改为在成就评估时动态触发（而非引导时一次性排期）
3. **两套 streak 统一或明确分场景** — 给用户清晰的解释差异
4. **moneySaved 添加除零保护** — 对齐 `totalSpentOnSmokes` 的 `max(1, ...)` 守卫
