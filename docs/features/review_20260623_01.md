# Review 01: SmokeFree 全量代码综合审查

Date: 20260623
Branch: N/A (全量审查，无 diff)

## Summary

对 SmokeFree iOS 控烟应用的 38 个 Swift 源文件 + 4 个测试文件进行了全量代码审查。代码整体架构清晰，MVVM 分层合理，SwiftData 使用规范。主要问题集中在三个方面：(1) 测试用例与 Phase 7 改版后的生产代码严重脱节，多个测试因引用已删除的里程碑/徽章 ID 而无法通过或无法编译；(2) 数值计算存在多处除零风险；(3) 错误处理普遍缺失。本次审查共发现 **4 个 Critical、7 个 Major、12 个 Minor** 问题。

## Issues

### Critical

- [C1] **UserProfileTests.moneySaved 调用签名错误 — 测试无法编译** — `UserProfileTests.swift:54,66,79`
  `moneySaved` 是实例方法 `func moneySaved(logs: [SmokingLog]) -> Double`，但测试中直接作为属性调用 `profile.moneySaved.isApproximatelyEqual(...)`，未传 `logs:` 参数。编译阶段即报错，3 个 moneySaved 测试用例全部失效。

- [C2] **DashboardViewModelTests 引用了 Phase 7 已删除的里程碑 ID** — `DashboardViewModelTests.swift:33,44,48,82,97`
  Phase 7 将里程碑从生物学恢复时间（20min、8hours 等）重写为控烟里程碑（day1、3days 等），但测试仍然断言 `nextMilestone?.id == "20min"`，且注释内容引用已不存在的"20 分钟""8 小时"里程碑。`update_25SecondsIn_firstMilestoneIsNext`、`timeRemaining_withinOneHour_showsMinutes`、`timeRemaining_withinOneDay_showsHours` 共 3 个测试用例必然失败。

- [C3] **AchievementServiceTests 引用不存在的徽章 ID** — `AchievementServiceTests.swift:142,161,177`
  测试断言 `reduction_first_day` 和 `reduction_3days` 徽章颁发，但 AppConfig.achievementDefinitions 中不存在这两个 ID。Phase 7 将减量类徽章从 `reduction_first_day/reduction_3days` 重命名为 `streak_1_day/streak_3_days` 等控烟坚持类徽章（条件也变为 `requiredConsecutiveDaysBelow`）。这导致 3 个测试用例失败（`awards_reductionFirstDay_whenTodayBelowBaseline`、`awards_reduction3Days_whenThreeConsecutiveDaysBelow`、`noReduction3Days_whenOnlyOneDayBelow`）。

- [C4] **AchievementServiceTests 所有 streak 类测试因 `logs: []` 而必然失败** — `AchievementServiceTests.swift:22-108`
  当前徽章 `streak_1_day`、`streak_3_days` 等的条件字段是 `requiredConsecutiveDaysBelow`（严格：需要实际日志记录），但 `awards_streak1Day_whenStreakIs1`、`awards_multipleStreakBadges_when7DayStreak`、`awards_allStreakBadges_when365DayStreak`、`doesNotRewAward_alreadyUnlockedBadge`、`persists_unlockedAchievements_toContext` 这 5 个测试全部传入 `logs: []`，导致 `consecutiveDaysBelow` 返回 0，无法达成任何 streak 徽章条件。共 5 个测试用例必然失败。

### Major

- [M1] **fatalError 暴露内部错误给用户** — `SmokeFreeApp.swift:29`
  `fatalError("Failed to create ModelContainer: \(error)")` 在 ModelContainer 初始化失败时会导致 App 崩溃，且错误描述包含技术细节。应使用用户友好的错误处理或至少降级为 `isStoredInMemoryOnly` 模式。

- [M2] **moneySaved 计算无除零保护** — `UserProfile.swift:74`
  `price / Double(perPack)` 中 `perPack` 来自 `log.cigarettesPerPackAtTime ?? cigarettesPerPack`，若用户将 profile.cigarettesPerPack 设为 0，则此处除以零。`updateReduction` 中 `pricePerPack / Double(profile.cigarettesPerPack)` 无保护，`updateCost` 虽有 `guard profile.cigarettesPerPack > 0` 但 `totalSpentOnSmokes` 计算路径中 `max(1, perPack)` 仅在一处使用，moneySaved 路径未受保护。

- [M3] **LoggingView 中成就评估使用陈旧数据** — `LoggingView.swift:26-28`
  `onSave` 回调中 `AchievementService.evaluateAndAward(profile: profile, logs: Array(logs), context: context)` 使用的 `logs` 是 `@Query` 在保存前的快照，不包含刚保存的今日记录。不过 DashboardView 的 `onAppear` 和定时器会重新评估，影响范围有限。

- [M4] **所有 SwiftData 写入操作缺少错误处理** — 全局
  `context.insert()`、`context.delete()` 无任何错误处理。SwiftData 虽在 context 关闭时自动保存，但如果保存失败（如 CloudKit 冲突），用户不会收到任何反馈，记录可能静默丢失。至少应在关键路径（保存日志、创建目标）记录错误日志。

- [M5] **EditBaselineView 无输入校验** — `DashboardView.swift:256-270`
  修改基准时，`newPrice` 可以是 0 或负数（TextField 使用 `.number` 格式，无下限限制），`newBaseline` 最小为 1（Stepper）。但如果用户通过键盘输入 0 或负数，pricePerPack 可被设为非法值，导致后续所有 moneySaved 计算错误。

- [M6] **GoalsViewModel.checkCompletion 当同时设天数+金额时行为与设计不符** — `GoalsViewModel.swift:31-44`
  代码中 `if let target = goal.targetMoneySaved` 优先检查金额，这意味着金额目标存在时天数条件被忽略。但 DESIGN.md 描述为"二选一"。同时 `GoalsViewModelTests.checkCompletion_eitherConditionSuffices` 测试了"任一条件满足即完成"的行为，与设计文档的"二选一"矛盾。需明确意图并统一。

- [M7] **HealthKitService 静默吞错误** — `HealthKitService.swift:62`
  `try? await store.save(sample)` 使用 `try?` 丢弃所有错误，包含授权失败、数据写入失败等。用户无法得知 HealthKit 集成是否正常工作。应至少 `Telemetry` / `os_log` 记录失败。

### Minor

- [m1] **货币计算使用 Double 而非 Decimal** — `UserProfile.swift:68`、`DashboardViewModel.swift:59,73,88`
  项目中所有货币字段（pricePerPack、totalCost、moneySaved）使用 Double 类型，浮点累加可能产生精度误差（如 ¥0.1 的多次累加）。虽 NumberFormatter 缓解显示问题，但项目规则明确规定"All monetary amounts must use int (cents) / Decimal"。对于个人追踪 App 影响有限，但建议长远修复。

- [m2] **等价物换算逻辑散落在 View 层** — `DashboardView.swift:97-103`、`ProgressTabView.swift:44-50`
  `ReductionProgressCard.equivalentText` 和 `ExpenseStatsView.bestComparison` 中的等价物列表和计算逻辑直接硬编码在 View 中，未提取为独立配置或 ViewModel 方法。DESIGN.md 中 costComparisonPresets 已在 AppConfig 定义，但这两处未复用。

- [m3] **NumberFormatter 每次调用重复创建** — `DashboardViewModel.swift:101-107,136-142`、`ProgressTabView.swift:54-61`、`SmokeFreeWidget.swift:63-69`、`EditGoalView.swift:117-123`
  多处每次格式化都新建 `NumberFormatter()` 实例。NumberFormatter 初始化开销较大，应缓存为 static 属性。

- [m4] **`actualStreakDays` 和 `consecutiveDaysBelow` 为 O(n×m) 复杂度** — `UserProfile.swift:48-65`、`AchievementService.swift:59-78`
  while/for 循环中嵌套 `logs.first(where:)` 线性查找，当日志量增长后性能下降。可用 Dictionary 按日期索引优化为 O(n)。

- [m5] **DashboardView Timer 持续运行** — `DashboardView.swift:74-76`
  `Timer.publish(every: 60, ...).autoconnect()` 在 TabView 中永不销毁，即使用户在其他 Tab 也每分钟触发更新。建议在 `onDisappear` 时取消订阅，或使用 ScenePhase 管理。

- [m6] **consecutiveDaysBelow 递减日期失败时的 fallback 逻辑有误** — `AchievementService.swift:75`
  `cal.date(byAdding: .day, value: -1, to: checkDate) ?? checkDate` — 若日期运算返回 nil（极端情况），保持 checkDate 不变，循环内重复检查同一日期 365 次，最终返回错误计数而非正确中断。

- [m7] **无数据空态下 Charts/Trends 无 empty state** — `TrendsView.swift`
  当无吸烟记录时，图表显示全零柱状图，无引导提示。DashboardView 有 "完成引导以开始" 但 Charts 无对应处理。

- [m8] **WidgetProvider.placeholder 数据为硬编码** — `WidgetProvider.swift:31-37`
  placeholder 数据为固定值（7 天、¥56），若用户实际为不同值，Widget 加载时会出现短暂的数据跳动。建议使用更具代表性的默认值（如 0 天、¥0）。

- [m9] **GoalsView 中 checkCompletion 每次 onChange 重新计算全量日志** — `GoalsView.swift:73-81`
  `onChange(of: goals)` 和 `onChange(of: logs)` 均触发 `checkCompletion()`，每次调用 `profile.actualStreakDays(logs:)` 和 `profile.moneySaved(logs:)`，在底层已有 O(n) 计算的情况下存在冗余调用。

- [m10] **OnboardingViewModel.canProceed 未校验 cigarettesPerPack** — `OnboardingViewModel.swift:15-23`
  Step 1 (SmokingHabitsView) 仅校验 `cigarettesPerDay > 0 && pricePerPack > 0`，未校验 `cigarettesPerPack`。虽 Stepper 下限为 10，但若未来修改 Stepper 范围或用户通过其他方式设为 0，会进入后续计算路径引发除零。

- [m11] **AchievementDefinition.requiredStreakDays 字段为死代码** — `AppConfig.swift:20,32`
  `requiredStreakDays` 在所有 11 个徽章定义中均未赋值（为 nil），但 `AchievementService.evaluateAndAward` 中仍检查此字段。可清理或保留用于未来扩展。

- [m12] **AchievementServiceTests.doesNotRewAward 方法名有 typo** — `AchievementServiceTests.swift:71`
  方法名为 `doesNotRewAward` 应为 `doesNotReAward`。

## Escalate

- [ESCALATE] 测试策略决策 — 当前 4 个测试文件共 32 个用例（按 TESTING.md），经审查发现至少 **11 个用例存在编译/逻辑错误**（[C1]-[C4]）。需决定：方案 A 是将测试更新为匹配当前生产代码（修改断言和测试数据），方案 B 是重写测试以反映 Phase 7 改版后的新语义。建议优先方案 A（最小修改），同时补充缺失的边界测试用例。

- [ESCALATE] GoalsViewModel.checkCompletion "二选一" vs "任一满足" 语义冲突 — DESIGN.md 说 `targetDays` / `targetMoneySaved` 是二选一，但代码实现为金额优先（设置金额则忽略天数），而测试 `checkCompletion_eitherConditionSuffices` 测试的是"任一条件满足即可"。需明确产品意图并统一文档和代码。
